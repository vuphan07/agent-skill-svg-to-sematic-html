# SVG export -> semantic HTML/CSS

Design tools export a whole screen as one giant `<svg>` (raster images embedded
as base64, text as `<text>/<tspan>`, shadows as `<filter>`, shapes as
`<rect>/<line>/<path>`). This skill converts that into HTML in two flavors and
verifies the result against the original.

## Decide the target first

Ask (or infer) which the user wants:

- **Pixel-perfect, flat** — every primitive absolutely positioned. Fast, looks
  identical at the fixed export width, but not semantic and not responsive.
  -> use `transpile_flat.py` and you are basically done.
- **Semantic / structured ("not flat", "proper HTML")** — real `<header>`,
  `<section>`, `<p>`, `<form>`, `<footer>`, document flow. Needs the analysis
  output + manual rebuild following `references/semantic-rebuild.md`.

Most "make it proper HTML" requests mean the **semantic** path. When in doubt,
produce the flat draft first (it's instant) and use it as a visual reference
while building the semantic version.

## Project layout & the check rule

Use two sibling folders and match files by **base name**:

```
svg-input/      # SOURCES: the .html (SVG-wrapped) / .svg exports to convert
html-output/    # RESULTS: the generated page per source, same base name
                #   html-output/<name>.html   (or html-output/<name>/index.html)
compare/        # generated by batch_check.py: comparison strips per source
```

So `svg-input/13.html` is the source of truth for `html-output/13.html`.

**The check rule (do this before calling any file "done"):** a generated file is
only accepted after a section-by-section visual match against its source. View
the source rendered as an image (serve it on localhost or let the headless
browser capture it), capture the generated page the same way, and compare them
band by band. Fix every section discrepancy — wrong icon, missing accent line /
chevron / separator, bold-looking text, wrong line breaks, off spacing — then
re-render and re-check. Only the **font-width height delta** (see caveat below)
is allowed to differ, because that comes from a missing Segoe UI, not the markup.

Manual viewing:    `python scripts/serve.py svg-input 5501`  -> open http://localhost:5501/svg-input/{name}.html
Automated check:   `python scripts/batch_check.py` (renders + stitches every matched pair)

## Workflow

### 1. Extract the embedded images
```bash
python scripts/extract_assets.py INPUT.html OUTDIR
```
Writes `OUTDIR/assets/imgN.*` (deduplicated) and prints which pattern/image id
maps to which file. **Open the PNGs and identify each one** (logo, hero photo,
phone screenshot, step icons, store badges) — you'll place them by meaning.

### 2. Measure everything
```bash
python scripts/analyze_svg.py INPUT.html --json OUTDIR/analysis.json
```
Prints DESIGN TOKENS (colors, font-sizes, line-gaps, radii, shadow, canvas),
the SPINE (every block with x / top / bottom / gap-to-previous), LINE BREAKS
(the tspan splits), and **FULL-BLEED LAYERS**. These are your exact numbers —
don't eyeball spacing.

**Read FULL-BLEED LAYERS first.** It lists every page/section background
(full-width rect/image: hero gradient, section tint panels) with its y-span and
**how many foreground blocks it bleeds behind**. A layer "behind 6 blocks
y[20,652]" means it spans 669px down *behind the first card/photo* — attach it to
the **page/section** at that height, never to a short hero element (the linear
SPINE alone hides this z-order, which is how a hero gradient gets clipped).

### 3a. Flat path
```bash
python scripts/transpile_flat.py INPUT.html OUTDIR
```
Produces `OUTDIR/index.html` + assets. Done.

### 3b. Semantic path
Read `references/semantic-rebuild.md` and build a **single `html-output/<name>/index.html`**
by hand using the analysis: tokens -> CSS variables, SPINE `gap` -> `margin-top`
between blocks, LINE BREAKS -> `<br>`, primitives -> the mapping table.

**Keep CSS and JS INSIDE the HTML** — put all styles in one `<style>` in `<head>`
and any behaviour in one `<script>` before `</body>`. Do **not** emit a separate
`style.css`. One self-contained file is the deliverable: it renders identically,
the verify loop reads it directly, and it's far easier to hand off into a UI/design
tool later. (Assets stay in `assets/` as files; embed them as base64 only when a
fully portable single file is needed — see §5.)

Key rules that bite people:
- SVG text is **regular weight** — use `<p>`, not `<h1/2/3>`, and set
  `font-weight:400` globally (heading tags/600 make text look wrongly bold).
- Reproduce the SVG's **hard line breaks** with `<br>`.
- Don't forget faint **accent/separator lines**, **chevrons**, **section tint
  panels**, and correct **icon identity/order**.
- **Letter-spacing is per-element, not global.** Display/heading text often uses
  a tighter value than body (e.g. `-0.06em` on the h1 vs `-0.015em` everywhere
  else). Grep `letter-spacing="..."` and apply per text role — a wrong value
  makes a heading look too wide *and* too heavy.
- **Tinted panels are usually a gradient/pattern painted at reduced `opacity`**
  (e.g. `fill="url(#pattern-2)" opacity="0.45"`), i.e. blended onto the page
  colour — NOT the raw extracted image at full strength. Reproduce with a
  page-coloured veil over the image (`linear-gradient(rgba(bg,.55),rgba(bg,.55)),
  url(grad.png)`). Reusing the raw gradient makes panels far too saturated.
  **The hero image often carries the SAME `opacity="0.45"`** — veil it too, don't
  paint the hero at full strength (a too-pink hero is the classic tell).
- **Pattern fills use `preserveAspectRatio="none"` → `background-size:100% 100%`
  (STRETCH), not `cover`.** `cover` crops to the image's most-saturated corner and
  reads too pink/dark; stretch shows the whole gradient like the export does.
  (`extract_assets.py` prints `mode:"stretch"` vs `"cover"` per pattern.)
- **Full-bleed hero gradients fade to the page colour** via an overlaid
  `fill="url(#linear-gradient)"` rect (white-transparent → bg). Reproduce the
  fade (`linear-gradient(transparent Ypx, bg Y2px)` overlay), don't hard-cut;
  read the rect's `transform` y for where the fade starts.
- **Put the top gradient on `.page`, not on a short hero element.** The gradient
  image spans the top ~669px of the whole page and **bleeds down behind the
  following content** (e.g. behind the first card/photo), fading out. If the hero
  block is short (just a logo + title) and you attach the gradient to it, it gets
  clipped to that ~140px and cuts to white abruptly. Attach the 669px gradient +
  fade to the page-level top background so it shows through the next section.
- **Segmented/toggle controls:** the selected segment is a separate inset rect
  (often `fill="#fff" opacity="0.8"`) floating on a darker track (a pattern /
  gradient rect). Check **which side** the inset pill is on and **its colour** —
  don't assume the dark segment is the active one. Give the track `padding` so
  the pill floats with the track showing around it.
- **Icon stroke/fill colour ≠ body ink.** Small icons (hamburger, chevron, `+`)
  are often pure `#000` or a distinct gray (`#3d3d3d`), not the text `#272825`.
  Read the colour off the icon's own `<line>/<path>`.
- When you map an icon/CTA to a `<button>`, **reset native chrome**
  (`background:none;border:none;padding:0`) or the browser's default button box
  appears (a hamburger renders as a boxed glyph otherwise).
- `analyze_svg.py` does **not** surface per-element `opacity`, distinct
  `letter-spacing`, `fill="url(#pattern)"`, or icon stroke colours — grep the
  raw SVG for these before finalizing CSS.
- **Reuse the SVG's OWN icon paths — don't hand-draw approximations.** Each
  icon in the export is a `<g id="Icon-N">` holding the real `<path>`/`<line>`s.
  Copy that geometry verbatim (compose the parent-`<g>` `transform` with the
  path's `transform` into one) and keep its exact `fill`/`stroke`. A redrawn
  checkmark/arrow/chevron/plus/social glyph never quite matches; the source path
  does. (e.g. the badge check is `id="Icon-27"` → one `<path fill="#9d1c1b">`;
  the button arrow is a white circle + a red chevron, not a lined arrow.)
- **Lines (separators / accents / underlines) are their own element, never a CSS
  `border`.** Render each as an `<hr>`/`<span>` with the SVG line's **exact**
  width and colour/opacity. Borders inherit the element's box width so they
  overflow the intended 353px content rule and can't reproduce the short fixed
  underlines (e.g. 61px / 67px under a link). In these exports every full-width
  rule is `<line x2="353" translate(20 Y)>` — i.e. 353px starting at x=20.
- **A "line" may be a `<line>`, a thin `<rect>`, OR a `<path>`** — vertical
  accents are often `<path d="M0,253V0">`. Inventory all three element types;
  grepping only `<line>` silently drops path-drawn rules (this is how a feature
  accent goes missing). Confirm by pixel-scanning the rendered source for
  dark columns/rows where you expect a rule.
- **Interactive controls need real behaviour, not just the look.** Tabs/segmented
  toggles need a small JS handler to switch (move the pill + swap text colour);
  radios/checkboxes must be real `<input>` so they're clickable. Read the
  selected-pill rect to get geometry right — it is usually **flush** (full track
  height, ~half width, same radius) not inset; don't invent padding insets.
- **Get block gaps from absolute coords, not the eye.** The space between an
  image/mockup and the element above it is `thisTop - prevBottom` in the SPINE
  (e.g. CTA button bottom → first hero badge top = 58px). Eyeballing the mockup
  offset is the usual cause of "image too close to / far from the button".
- **Make every inline icon self-contained — don't rely on CSS for its size or
  colour.** Each inline `<svg>` must carry `xmlns="http://www.w3.org/2000/svg"`,
  explicit `width`/`height` attributes, and explicit `fill`/`stroke` values
  (avoid `currentColor`). Draw small UI marks — hamburger, radio dot, checkbox,
  chevron, plus, social — as inline SVG with those attributes, **not** as
  CSS-bordered `<span>`s or `::before/::after` pseudo-elements. Why: many places
  the file actually gets opened — print/Save-as-PDF, "export as image",
  paste into a design tool (Figma/XD), email/WebView, a strict XML/XHTML parser,
  or the packaged single-file build — **drop or ignore CSS applied to inline SVG
  and CSS borders/pseudo-elements**. The icons then vanish, or (being unsized)
  collapse and float to the wrong place — even though they look perfect in a
  desktop browser / the puppeteer screenshot. A CSS class for sizing is fine as
  a convenience, but the SVG must still render correctly with the class stripped.

### 4. Verify section by section (the check rule)
Put sources in `svg-input/`, generated pages in `html-output/` (same base name), then:
```bash
python scripts/batch_check.py            # render + stitch + DETERMINISTIC diff, every pair
python scripts/batch_check.py --ai       # also send flagged crops to Gemini (vendored gemini_vision.py)
# or one pair manually:
node   scripts/render_compare.js svg-input/13.html html-output/13/index.html compare/13
python scripts/visual_diff.py compare/13/original.png compare/13/candidate.png --out compare/13
```
Open `compare/<name>/compare_*.png` (source left, your build right). Fix every
section discrepancy and re-run. A file is "done" only when each section matches
(ignoring the font-width height delta). To eyeball a source in the browser:
`python scripts/serve.py svg-input 5501`.

**Trust the deterministic diff first; your eyes and the AI second.** `visual_diff.py`
writes `compare/<name>/diff_report.json` + annotated crops, and is the reliable
judge for the classes eyeballing keeps missing:
- **tintΔ** (per-strip RGB delta on proportionally-resampled pages) → catches
  over-saturated gradients / wrong panel tint / missing fills. *This caught two
  real bugs a human review missed: a hero painted at full strength instead of
  `opacity:.45`, and cards using `cover` instead of stretch.*
- **rule tally** (src vs cand count of full-width rules) → catches missing
  separators/accents.
- It is **drift-robust on purpose** (no row registration) — so it will NOT flag a
  1–2px offset; use the native-res crops below for those.
Read the JSON, fix every HIGH, eyeball every MED. `--ai` adds Gemini findings per
flagged crop, but treat them as **leads to verify, not facts**: Gemini also reports
font-fallback noise, speculative "missing shadow/radius", and crop-misalignment
artifacts. The deterministic metric decides; the AI only helps name *what* differs.

**"Matches at a glance" is not enough.** The stitch strips are downscaled, so
subtle-but-obvious diffs (heading width/weight, gradient saturation, toggle
inset side, hamburger/chevron shape, icon colour) hide. Render both, then build
**native-resolution side-by-side crops** of the suspicious regions — header,
each toggle, each tinted panel, the icon clusters — and inspect those. E.g.:
```python
from PIL import Image
o=Image.open("compare/13/original.png"); c=Image.open("compare/13/candidate.png")
def sbs(y0,y1,name):           # original | candidate, no downscale
    oc,cc=o.crop((0,y0,o.width,y1)),c.crop((0,y0,c.width,y1))
    cv=Image.new("RGB",(oc.width+cc.width+16,y1-y0),(255,0,255))
    cv.paste(oc,(0,0)); cv.paste(cc,(oc.width+16,0)); cv.save(name)
sbs(0,360,"compare/13/z_header.png")     # header + h1
```
Loop: read SVG attrs → predict → render → zoom-compare → fix → repeat, region
by region. Confirm a fix against the SVG source AND the screenshot, not one
alone.

**The puppeteer render is a full desktop Chrome — it passes things that break in
the real delivery target.** Inline-SVG icons sized/coloured only via CSS,
CSS-bordered radio dots, and `::before/::after` marks look perfect here yet
vanish or mis-place when the file is printed/Saved-as-PDF, exported to image,
pasted into a design tool, shown in a WebView/email, or run through a strict
parser. So also (a) build the packaged single-file `.html` and open *that*, and
(b) open the page in the viewer the user will actually use. If an icon is
missing/displaced there but fine in puppeteer, apply the self-contained-SVG rule
(§3b) — don't dismiss it as "works for me". Re-check specifically: the header
hamburger, every button arrow-circle, link chevrons, FAQ `+`, radio dots, and
the hero badge checks.

### 5. Package (optional) — embed assets for a portable single file
CSS and JS already live inside `index.html` (§3b), so it is self-contained except
for the linked images in `assets/`. That's the normal deliverable. When you need
**one file with nothing external** (e.g. to email or paste into a design tool),
base64-embed the images too:
```bash
python scripts/inline_build.py html-output/13 --embed-assets --out html-output/13/13.standalone.html
```
`inline_build.py` base64-inlines `assets/*` (and, defensively, folds in a
`style.css` if one still exists). Without `--embed-assets` it's a no-op for an
already-inline page. Verify with `visual_diff.py` first; embedding doesn't change
rendering. Or just zip `index.html + assets/`.

### 6. Learn from every bug (keep this skill getting smarter)
**Whenever a discrepancy is found — a user bug report in `bugs/<name>/` (e.g.
`current*.png` vs `expect*.png`), or one you catch in verify — do NOT just patch
the pixel. Run this loop so the same class of bug can't recur:**
1. **Root-cause it.** Ask *which step let this through* — was it missed in
   `analyze_svg.py` (not surfaced), in the build (rule not followed), or in verify
   (flagged but dismissed)? Name the step.
2. **Fix the affected output.**
3. **Encode the lesson at the STRONGEST layer possible** (mechanise > prose):
   - If a script *could* have detected it → add/extend a detector in
     `analyze_svg.py` (preferred) or a metric in `visual_diff.py`, so the next run
     is *told* about it automatically. (e.g. the clipped-hero-gradient bug → added
     the FULL-BLEED LAYERS report.)
   - Otherwise → add one concise bullet to "rules that bite people" (§3b) as
     **symptom → cause → fix**.
4. **Append a one-liner to the CHANGELOG below** so the knowledge accrues and is
   visible at a glance.
A fix that only repairs one file but doesn't teach the skill is incomplete.

#### CHANGELOG — bugs turned into rules/detectors
- Heading too wide/heavy → per-element `letter-spacing` (h1 `-0.06em`), §3b rule.
- Panels/hero too saturated → gradient/pattern at `opacity` + `stretch` not
  `cover`; veil rule, §3b.
- Icons vanish/misplace in PDF/design-tool/strict parser → self-contained inline
  SVG (xmlns + explicit w/h + colours), §3b.
- Hand-drawn vs source icons; lines as components not borders; a "line" may be
  `<path>` → §3b rules + pixel-scan check.
- Hero gradient clipped to a short hero → **FULL-BLEED LAYERS** detector in
  `analyze_svg.py` + "attach top gradient to .page" rule, §3b.

## Dependencies
- Python 3 (stdlib). `Pillow` + `numpy` (`stitch.py`, `visual_diff.py`):
  `pip install pillow numpy`.  (No scikit-image needed.)
- Node + `puppeteer` for rendering (`npm i puppeteer`); set `CHROME_PATH` to
  reuse an existing Chrome/Chromium.
- **Optional AI layer** (`--ai`): `pip install google-genai` + a `GEMINI_API_KEY`
  (env, or a `.env` in this skill dir / project root / `~/.claude/.env` — see
  `.env.example`). This is **self-contained** — `scripts/gemini_vision.py` is a
  ~60-line vendored Gemini vision client, so the skill needs NO external
  multimodal skill. The whole verify pipeline runs offline without a key; `--ai`
  only adds Gemini's semantic labelling on already-flagged crops.

## Font caveat (always mention to the user)
These exports target **Segoe UI**. Without that font installed, the fallback is
wider, so text wraps differently and the page grows taller even when the CSS is
correct. Keep `'Segoe UI'` first in the font stack; for cross-platform fidelity
embed a metric-compatible font (e.g. Microsoft Selawik, or a licensed Segoe UI
woff2). See `references/semantic-rebuild.md` §7.
